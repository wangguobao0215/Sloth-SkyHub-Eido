# 安全、治理与合规

> 本文档定义 SkyHub 技能族在数据安全、自动化治理、优雅降级、失败处理及合规部署方面的完整规范。
> 所有 Skill 开发与运维必须遵守本文档中的分类标准与审批流程。

---

## 1. 数据分类（Data Classification S1-S4）

| 等级 | 名称 | 定义 | 示例 | 处理规则 | 存储要求 |
|------|------|------|------|----------|----------|
| S1 | 公开 (Public) | 公开渠道可获取的信息 | 公司名称、行业、公开新闻、官网信息 | 无限制 | 正常存储 |
| S2 | 内部 (Internal) | 企业内部使用的运营数据 | 健康分、Playbook 状态、CSM 备注、内部标签 | 仅内部人员可访问 | Obsidian Vault，基于角色权限控制 |
| S3 | 敏感 (Confidential) | 涉及商业机密的数据 | 合同金额、折扣率、联系人直线、续约条件 | 使用前脱敏，限定授权人员 | 加密存储，frontmatter 字段脱敏 |
| S4 | 高度敏感 (Restricted) | 受法律法规保护的数据 | 身份证号、银行账号、个人健康信息 | 禁止进入知识库 | 永不存储于 Obsidian，仅保留在源系统 |

### 分类判定规则

- **默认等级**: 新数据字段未显式标注前，默认视为 S2（内部）
- **就高原则**: 一条记录包含多个等级字段时，整条记录按最高等级处理
- **继承规则**: 衍生数据（汇总、均值）的等级不低于源数据中的最高等级
- **降级审批**: S3 降为 S2 需 CS Manager 审批；S4 不可降级

### 脱敏规则（Masking Rules）

**LLM 输入脱敏** -- 数据传入 LLM 之前必须执行：

| 原始字段 | 脱敏方式 | 脱敏后示例 |
|----------|----------|------------|
| 合同金额 | 替换为级别标签 | `[ARR-Band: 50w-100w]` |
| 手机号 | 掩码处理 | `138****6789` |
| 邮箱 | 掩码处理 | `zhang***@company.com` |
| 折扣率 | 替换为类别 | `[折扣类型: 标准]` 或 `[折扣类型: 特殊]` |
| 联系人姓名 | 保留姓氏 | `张*` |
| 公司内部代号 | 使用匿名 ID | `Account-A0372` |

**跨角色查看**: CSM 看自己客户完整 S2/S3 数据，看不到其他 CSM 的 S3 字段（合同金额、折扣率自动脱敏）；CS Manager 可看团队全部 S2/S3；CS VP 可看全部数据但 S4 仍不进入知识库。

**报告输出**: 对外报告排除内部成本/利润信息；案例库使用范围值（如"年合同额50-100万"）和行业代称；数据导出 S3 及以上自动标记 `[CONFIDENTIAL]` 水印。

## 2. 自动化级别（Automation Level L1-L5）

| 级别 | 名称 | AI 角色 | 人类角色 | 审批要求 | 典型场景 |
|------|------|---------|----------|----------|----------|
| L1 | 信息收集 (Collect) | 数据搬运工 | 读者 | 无 | 健康报告数据展示、仪表盘刷新 |
| L2 | 分析建议 (Advise) | 参谋 | 决策者 | 无 | 流失预测、竞对分析、健康评分 |
| L3 | 草稿生成 (Draft) | 文案 | 审核人+发送者 | CSM 确认 | 企微话术、QBR 报告、邮件模板、价值报告 |
| L4 | 自动执行 (Execute) | 执行者 | 抽查者 | Team Lead 预授权+事后审计 | Tech-touch 自动邮件、Onboarding 提醒 |
| L5 | 全自动闭环 (Autonomous) | 全权代理 | 监督者(异常介入) | CS VP 审批+季度审计 | 目前不开放，仅超低风险场景 |

### 级别决策树

```
Q1: 输出是否对外可见？
├── 否 -> 是否包含 AI 判断？ 否=L1 / 是=L2
└── 是 -> Q2: 发送前需人工审核？
          ├── 是 -> L3
          └── 否 -> Q3: 有事后抽查？ 是=L4 / 否=L5(需VP特别审批)
```

### 级别变更流程

**升级条件**: 当前级别运行满 30 天 + 准确率/采纳率 >= 95% + 无 P0/P1 事故 + CS VP 书面审批。

**降级条件**: CS Manager 即时执行（安全优先，无需等待审批）。触发条件：准确率 < 85%、客户投诉、数据安全事件。降级后 7 天内复盘。

所有变更记录于 `_governance/automation_level_changelog.md`：

```yaml
- date: 2025-07-15
  skill: health-report
  change: L2 -> L3
  reason: "健康报告摘要质量达标，CSM采纳率97%"
  approver: "李总 (CS VP)"
```

## 3. 优雅降级（Graceful Degradation G1-G3）

| 级别 | 名称 | 策略 | 原则 |
|------|------|------|------|
| G1 | 轻度 (Mild) | 使用缓存数据或简化计算逻辑 | "略旧可接受" |
| G2 | 中度 (Moderate) | 使用预设模板或通用建议替代个性化内容 | "通用优于空白" |
| G3 | 重度 (Severe) | 仅返回原始数据，CSM 自行判断 | "人工兜底最坏情况" |

### 设计原则

1. **"有损降级优于完全失败"** -- 始终返回有用内容，绝不返回空白
2. 降级路径 G1 -> G2 -> G3 逐步后退，不允许跳级
3. 降级输出必须在头部显式标注降级级别和原因
4. 所有降级事件自动写入 `_governance/degradation_log.md`，月度复盘
5. 降级后持续探测原始能力是否恢复，恢复后自动升回

### 降级通知模板

```markdown
> [!warning] 降级通知 (Degradation: G2)
> 数据源 [CRM API] 响应超时，已切换为通用模板模式。
> 以下为标准化内容，非个性化分析。预计恢复: 2小时内。
```

## 4. 五大失败场景与应对策略

### 场景一：LLM 幻觉（Hallucination）

- 关键事实（金额、日期、人名）与知识库/CRM 原始数据交叉验证
- 输出标注置信度: `[高置信]` / `[中置信]` / `[低置信-请核实]`
- L3 强制审核: 所有对外内容必须 CSM 确认
- 建立"幻觉案例库"，持续优化 Prompt 和 Few-shot 示例
- 幻觉率 > 15% 时自动降级为纯检索模式（RAG without generation）

### 场景二：数据缺失（Missing Data）

| 缺失率 | 处理策略 |
|--------|----------|
| < 10% | 继续执行，缺失字段标注 `[数据待补充]` |
| 10%-30% | 继续执行，输出标注"部分数据缺失，仅供参考" |
| 30%-60% | 降级为 G2，标注 `[数据严重不完整]` |
| > 60% | 拒绝执行，返回缺失字段清单 |

执行前检查: 核心字段完整性扫描 + 数据时效性检查 + 跨源一致性比对。

### 场景三：API 超时（API Timeout）

1. **重试**: 最多 2 次，Exponential Backoff（5s -> 10s）
2. **缓存降级**: 非实时数据使用本地缓存（健康分 TTL=24h, 使用数据=4h, 合同=7d）
3. **备用通道**: 主 LLM 超时切换备用 LLM；主 CRM API 超时使用 Obsidian 缓存
4. **全面中断**: 返回最近缓存+时间戳标注，触发 G3 降级，通知 CS Ops

### 场景四：CSM 不信任（Trust Deficit）

- 按 CSM/Skill/客户维度监控采纳率，低于 60% 触发预警
- 展示 AI 推理过程和数据来源，不只呈现结论
- 采纳率持续低于 50%: 从推送模式切换为按需拉取
- 降低通知频率，合并低优先级提醒，减少侵入感
- 提供"有用/没用"快捷反馈，持续优化

### 场景五：客户质疑 AI（Customer Pushback）

- 所有外发内容以 CSM 个人名义，语气匹配 CSM 习惯
- L3 审核确保内容自然/个性化，去除 AI 痕迹
- 统一话术: "我们使用数据分析工具辅助，以便更好地为您服务"
- 客户明确不适 -> 立即切换纯人工模式
- 事后记录反馈，评估是否调整该客户自动化级别

## 5. 国企 / 信创合规（Localization & Compliance）

### 三种部署模式

| 模式 | LLM 位置 | 数据出境 | 适用客户 |
|------|----------|----------|----------|
| Mode A | 本地私有化（Qwen-72B、ChatGLM） | 无 | 政府、军工、金融核心 |
| Mode B | 国产云（通义千问/文心/DeepSeek） | 不出境（国内云） | 国企、央企、信创要求 |
| Mode C | 国际云（OpenAI/Claude） | 出境（需评估） | 外企、民企、无信创要求 |

### 模式选择

```
Q1: 有信创/数据主权要求？
├── 是 -> 数据不离开内网？ 是=Mode A / 否=Mode B
└── 否 -> 有出境合规顾虑？ 是=Mode B / 否=Mode C
```

### 合规要求

| 要求项 | Mode A | Mode B | Mode C |
|--------|--------|--------|--------|
| 完整审计日志 | 必须 | 必须 | 必须 |
| 信创认证组件 | 必须 | 推荐 | 不要求 |
| 数据安全白皮书 | 必须 | 必须 | 推荐 |
| 等保 2.0 | 按需 | 按需 | 通常不要求 |
| 个人信息保护影响评估 (PIA) | 必须 | 必须 | 推荐 |
| API 调用加密 | TLS 1.2+ | TLS 1.2+ | TLS 1.2+ |

### 审计日志规范

每次 AI 调用必须记录：

```yaml
- call_id: "uuid-v4"
  timestamp: "ISO-8601"
  skill: "health-report"
  automation_level: "L3"
  data_classification: "S2"
  input_masked: true
  llm_provider: "qwen-72b-local"
  operator: "csm-zhangsan"
  account: "Account-A0372"
  action: "generate_qbr_draft"
  output_reviewed: true
  reviewer: "csm-zhangsan"
```

## 6. 治理制品（Governance Artifacts）

| 制品 | 路径 | 更新频率 | 责任人 |
|------|------|----------|--------|
| 自动化级别变更日志 | `_governance/automation_level_changelog.md` | 每次变更时 | CS Ops |
| 降级事件日志 | `_governance/degradation_log.md` | 实时写入，月度复盘 | CS Ops |
| 风险仪表盘 | `_governance/risk_dashboard.md` | 每周更新 | CS Manager |
| Skill 审计日志 | `_governance/skill_audit_log.md` | Skill 上线/变更时 | Skill 开发者 |
| 数据分类登记表 | `_governance/data_classification_registry.md` | 新字段引入时 | CS Ops + 法务 |
| 合规检查报告 | `_governance/compliance_review.md` | 季度 | CS VP + 法务 |

### 月度治理复盘 Checklist

- [ ] 降级事件统计与根因分析
- [ ] 幻觉率趋势与 Prompt 优化效果
- [ ] 各 Skill 采纳率与 CSM 反馈汇总
- [ ] 数据缺失率趋势与数据质量改进
- [ ] API 可用性 SLA 达成情况
- [ ] 自动化级别变更回顾
- [ ] 安全事件与合规偏差检查

### 季度合规审计要点

1. **数据分类合规**: 抽查 Obsidian Vault，确认无 S4 数据泄漏
2. **自动化级别合规**: 验证所有 Skill 运行在审批后的级别
3. **脱敏有效性**: 抽查 LLM 调用日志，确认脱敏规则执行到位
4. **审计日志完整性**: 验证日志无缺失、无篡改
5. **降级机制有效性**: 模拟故障，验证降级路径正常触发
