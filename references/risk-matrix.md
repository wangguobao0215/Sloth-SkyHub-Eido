# 风险矩阵与缓解措施

> SkyHub CSM AI 系统实施过程中的十大风险识别、评估、缓解方案与监控机制。

---

## 十大风险

### R1: CSM 团队抗拒变革

| 属性 | 值 |
|------|-----|
| **类别** | 组织变革 |
| **影响** | 高 -- 直接决定项目成败，无人使用则投资归零 |
| **概率** | 高 -- 任何 AI 工具落地都会面临的首要挑战 |
| **风险属主** | CS VP |

**缓解措施:**

1. 制作 "AI 辅助而非替代" FAQ 文档，逐条回应 CSM 常见顾虑（如 "会不会取代我的工作"）
2. 先用 Champion 做成功示范，用真实案例（而非理论）说服团队
3. 初期考核仅奖不罚：使用 AI 产出好结果的加分，不使用的不扣分
4. CS VP 在团队会议中明确表态支持，定期参与 Skill 演示

**预警指标:**
- M3 后采纳率仍 < 30%
- 内部调研中出现系统性负面反馈（满意度 < 3.0）
- Champion 用户流失 >= 2 人

---

### R2: 数据质量不足

| 属性 | 值 |
|------|-----|
| **类别** | 数据治理 |
| **影响** | 高 -- 垃圾进垃圾出，健康分和预测完全失效 |
| **概率** | 高 -- 多数企业 CRM 数据质量堪忧 |
| **风险属主** | CS Ops |

**缓解措施:**

1. M1 启动数据清洗冲刺：集中 2 周修复核心字段（客户名、ARR、合同日期、CSM 归属）
2. 设定质量红线：核心字段缺失率 < 10%，否则该客户不纳入健康分计算
3. 开发数据自动补全 Skill：从邮件签名、CRM 备注、历史工单中提取缺失信息
4. 每条数据附带完整度置信标签（High / Medium / Low），供 CSM 判断可信度

**预警指标:**
- 核心字段缺失率 > 20%
- 健康分被 CSM 频繁手动覆盖（覆盖率 > 30%）
- 数据导入后 CRM 与 Vault 不一致记录 > 15%

---

### R3: LLM 幻觉导致客户事故

| 属性 | 值 |
|------|-----|
| **类别** | 技术风险 |
| **影响** | 高 -- 一次严重幻觉可能导致客户流失或法律风险 |
| **概率** | 中 -- 有知识库约束但无法完全消除 |
| **风险属主** | Tech Lead |

**缓解措施:**

1. L3 级别（人机协作）强制要求 CSM 审阅后再发送，系统不自动发出客户沟通
2. KB 交叉验证：AI 输出必须标注引用来源，无来源的回答标记为 "未验证"
3. 所有 AI 生成内容添加 "AI 辅助生成" 水印标记
4. 建立 2 小时事故响应 SLA：发现幻觉 -> 修正 -> 通知客户 -> 复盘

**预警指标:**
- 幻觉率 > 5%（抽检 AI 输出与事实不符的比例）
- 客户投诉 AI 生成内容不准确
- KB 命中率持续下降（< 80%）

---

### R4: 上线后采纳率低

| 属性 | 值 |
|------|-----|
| **类别** | 运营风险 |
| **影响** | 中 -- 系统能用但没人用，ROI 无法体现 |
| **概率** | 中 -- 与变革管理执行力强相关 |
| **风险属主** | CS Ops + PM |

**缓解措施:**

1. 将 Skill 嵌入现有工作流（如 CRM 侧边栏、Slack Bot），降低使用门槛
2. 每周发布 "AI 助攻榜"：展示谁用 AI 节省了最多时间、产出了最好结果
3. 设置 "每日推荐"：根据 CSM 当天日程主动推送可用 Skill
4. 建立反馈快速响应机制：CSM 提出的改进建议 48 小时内给出响应

**预警指标:**
- M6 后全员采纳率仍 < 40%
- 周活跃用户数连续 3 周下降
- Skill 调用集中在 1-2 个，其他 Skill 几乎无人使用

---

### R5: 国企/信创合规挑战

| 属性 | 值 |
|------|-----|
| **类别** | 合规风险 |
| **影响** | 高 -- 可能阻断特定客户群的服务 |
| **概率** | 低 -- 仅影响国企客户，且有替代方案 |
| **风险属主** | CTO + Legal |

**缓解措施:**

1. 提前调研信创名录，确认可用的国产 LLM（如通义千问、文心一言、GLM）
2. 架构设计支持 LLM 热切换：通过 MCP 抽象层隔离模型依赖
3. 数据本地化部署方案：私有化 Vault + 本地模型推理
4. 法务团队提前审阅数据处理协议（DPA），准备国企合规版本

**预警指标:**
- 国企客户明确要求数据不出境
- 信创政策收紧，影响已采用的技术栈
- 合规审查周期超过预期（> 2 个月）

---

### R6: 健康分模型脱离现实

| 属性 | 值 |
|------|-----|
| **类别** | 模型风险 |
| **影响** | 中 -- 健康分不准会削弱团队信任 |
| **概率** | 中 -- 初始权重基于经验，需数据校准 |
| **风险属主** | Data Analyst + CS Ops |

**缓解措施:**

1. 初始模型明确标注为 "Beta"，鼓励 CSM 反馈不一致之处
2. 每季度用实际续约/流失数据回归校准权重
3. 支持 CSM 手动调整分数并记录原因，作为校准输入
4. 设置置信区间：新客户或数据不足的客户健康分标注为 "低置信度"

**预警指标:**
- CSM 手动覆盖率 > 30%
- 健康分与实际续约结果相关性 < 0.5
- 同一客户健康分短期大幅波动（非业务原因）

---

### R7: 关键人员流失

| 属性 | 值 |
|------|-----|
| **类别** | 人员风险 |
| **影响** | 中 -- 核心知识集中在少数人，流失影响进度 |
| **概率** | 中 -- AI 人才市场竞争激烈 |
| **风险属主** | PM + HR |

**缓解措施:**

1. 所有架构决策、Skill 设计文档化到 Obsidian Vault，降低 bus factor
2. 关键角色设置 backup：Tech Lead 配 1 名副手，PM 配 1 名 CS Ops 协助
3. 知识转移会议：每月录制 1 次架构/设计走查，存档供新人学习
4. 与 HR 协作制定留才方案：项目奖金、技术成长路径

**预警指标:**
- 核心成员表达离职倾向
- backup 人员未到位超过 1 个月
- 文档更新频率下降（> 2 周无更新）

---

### R8: MCP 外部系统集成复杂度

| 属性 | 值 |
|------|-----|
| **类别** | 技术风险 |
| **影响** | 中 -- 集成失败会阻塞数据流，影响 Skill 效果 |
| **概率** | 高 -- 外部系统 API 变更、限流、权限问题频发 |
| **风险属主** | Tech Lead |

**缓解措施:**

1. 优先集成最关键的 2-3 个系统（CRM + 工单 + 邮件），其余按优先级排期
2. 每个集成设计降级方案：API 不可用时自动切换为手动导入模式
3. 建立 API 健康监控：响应时间、错误率、限流次数实时告警
4. 封装统一的 MCP Adapter 层：外部 API 变更只需修改 Adapter，不影响 Skill

**预警指标:**
- API 错误率 > 5%
- 集成开发耗时超过估算 50%
- 外部系统通知 API 即将废弃或重大变更

---

### R9: 项目 ROI 难以量化

| 属性 | 值 |
|------|-----|
| **类别** | 商业风险 |
| **影响** | 中 -- 无法证明价值会导致预算削减或项目中止 |
| **概率** | 中 -- CSM 效率提升的归因较难量化 |
| **风险属主** | PM + CS VP |

**缓解措施:**

1. Phase 1 启动前建立基线数据：QBR 准备时间、交接耗时、邮件响应时间等
2. 设计 A/B 对照：Champion 组 vs 非 Champion 组的效率对比
3. 多维度量化：时间节省（小时）、NRR 提升（百分点）、CSAT 变化
4. 每季度输出 ROI 报告，用 Value-Proof-Generator 自动生成

**预警指标:**
- 基线数据未在 M1 内收集完成
- 管理层对项目价值产生质疑
- 预算审批流程延长

---

### R10: 过度依赖 AI 导致技能退化

| 属性 | 值 |
|------|-----|
| **类别** | 长期风险 |
| **影响** | 低 -- 短期内不显现，长期影响团队能力 |
| **概率** | 低 -- 当前 L3 级别仍要求人工审核 |
| **风险属主** | CS VP + Training Lead |

**缓解措施:**

1. 保留 "无 AI 日"：每月 1 天纯手动操作，保持基础技能
2. AI 输出始终展示推理过程，CSM 需理解而非盲从
3. 培训课程持续包含基础技能模块（如客户沟通、谈判技巧）
4. 定期轮岗复杂场景：确保 CSM 能在 AI 不可用时独立处理

**预警指标:**
- CSM 无法解释 AI 建议背后的逻辑
- AI 系统宕机时团队效率断崖式下降（> 50%）
- 新入职 CSM 过度依赖 AI，基础技能评分下降

## 优先级矩阵

| 优先级 | 影响 x 概率 | 风险 | 处置策略 |
|--------|------------|------|----------|
| 立即行动 | 高 x 高 | R1 (变革抗拒), R2 (数据质量) | 专项攻坚，每周跟踪 |
| 重点关注 | 高 x 中 | R3 (LLM 幻觉) | 技术防线 + 流程兜底 |
| 持续监控 | 高 x 低 | R5 (信创合规) | 提前预研，按需激活 |
| 计划治理 | 中 x 高 | R8 (集成复杂度) | 架构解耦 + 降级方案 |
| 定期检查 | 中 x 中 | R4 (采纳率低), R6 (模型偏差), R9 (ROI 量化) | 季度审视，数据说话 |
| 预案准备 | 中 x 低 | R7 (人员流失) | 文档化 + backup 机制 |
| 长期观察 | 低 x 低 | R10 (技能退化) | 纳入培训体系，定期评估 |

### 矩阵可视化

```
         │  低概率      中概率       高概率
─────────┼──────────────────────────────────
 高影响  │  R5          R3          R1, R2
─────────┼──────────────────────────────────
 中影响  │  R7          R4,R6,R9    R8
─────────┼──────────────────────────────────
 低影响  │  R10         -           -
```

## 风险监控机制

### 月度风险评审会

- **主持:** CS VP
- **参与者:** PM, Tech Lead, CS Ops, 各风险属主
- **频率:** 每月第一个周五
- **议程:**
  1. 各风险属主汇报预警指标变化
  2. 新增/关闭风险讨论
  3. 缓解措施执行进度检查
  4. 升级事项决策

### 风险看板

- 实时更新在 Obsidian Vault 的 `_meta/risk-dashboard` 页面
- 每条风险含：当前状态（绿/黄/红）、最近更新日期、下一步行动
- Dataview 自动聚合预警指标数据

### 自动升级机制

| 触发条件 | 升级对象 | 响应时间 |
|----------|----------|----------|
| 任一风险变为红色 | CS VP + PM | 24 小时内 |
| 2 个以上风险同时变黄 | PM | 48 小时内 |
| 客户事故（幻觉/数据泄露） | CTO + CS VP | 2 小时内 |
| 采纳率连续 4 周下降 | CS VP + Training Lead | 1 周内 |

### 季度头脑风暴

- 每季度末进行一次 "Pre-mortem" 演练
- 假设项目已失败，逆向推导可能原因
- 更新风险清单，识别盲区
- 邀请外部顾问参与（可选）
